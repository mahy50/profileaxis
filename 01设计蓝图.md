# AI 驱动的 3D 铝型材网页设计系统落地蓝图

最优首版路线很明确：**不要把它做成 Web CAD，而要做成“受约束的会话式装配设计器”**。产品主链路应固定为：**自然语言意图 → 确认层 → 结构草案 DSL → 规则合法化 → 前端本地参数化建模 → 设计 BOM → 交易 BOM / 采购映射 → PDF 方案单**。这样做的根本原因是，型材系统本身就是按**型材系列、连接方式、加固件、脚杯 / 脚轮、附件**组织的模块化体系，供应商提供的技术数据与选型工具也围绕这些模块展开；载荷/挠度工具同样以型材型号、长度、载荷和支撑形式为输入，因此首版更适合“规则化装配 + 经验检测”，并不适合“自由几何 + 精细仿真”。citeturn22view3turn22view4turn18view6turn23view1turn18view7turn23view3

3D 引擎选择上，**首推 Babylon.js，Three.js 作为可选备份**。原因不是渲染性能，而是**编辑器建造成本**：Babylon.js 原生就给出 TransformNode、Gizmo、PointerDragBehavior、BoundingBoxGizmo、Utility Layer、Selection Outline、GUI 和 glTF 导出能力，且官方明确说明 Selection Outline 支持 instanced 与 thin-instanced meshes；Three.js 当然也能完成同类工作，但更依赖 Raycaster、TransformControls、OutlinePass、CSS2DRenderer 与若干 addon 的组合，而且 CSS2DRenderer 官方明确写了只支持 100% 浏览器与显示缩放，这会直接影响尺寸标注和文字标注的一致性。基于这些官方能力，Babylon.js 更适合首版“可编辑、可点选、可局部修改”的装配式 3D 应用。citeturn28search3turn28search0turn28search1turn28search2turn27search3turn27search4turn2search3turn26view1turn26view0turn26view2turn18view1turn26view3

## 产品定义与交互蓝图

### 需求分析与产品定义

这个项目不应被定义为“AI 帮用户画一个架子”，而应定义为**面向 DIY 用户的受约束装配系统**。它的真实产品形态由四层组成：**AI 意图层**负责理解自然语言；**规则约束层**负责把意图收敛为可装配结构；**参数化建模层**负责把合法结构实时变成 3D 方案；**交易映射层**负责把设计规格转换为采购项。这个分层正好对应工业语义里的 eBOM/mBOM 思路：设计侧保留“按设计表达的结构”，交易与制造侧再重组为采购/制造视图，而不是一开始就把草案绑死到商城 SKU。citeturn22view0turn22view1turn22view2

首版的真正目标，不是“理解一切货架需求”，而是**把高频、正交、矩形框架类置物架做深做透**。多家型材系统厂商的公开目录都在强调：其模块化型材最擅长的是机架、工作台、货架、围栏、推车这类由直线型材、直角连接件、地面构件和加固件组成的结构；这意味着你的首版边界应该明确收敛到**正交矩形框架 + 常见层板支撑 + 常见加固 + 常见地面构件**，而不是支持斜角、曲线、异形切口、任意孔位或自由草图。citeturn22view3turn22view4turn18view6turn23view1

首版建议把用户价值拆成三个核心结果：**能生成、能改、能下单准备**。生成指的是把自然语言需求稳定落到结构草案；能改指的是围绕选中对象做自然语言局部编辑；能下单准备指的是输出设计 BOM、采购清单映射和 PDF 方案单。这里面最容易被忽略的隐性需求，不在“AI 是否聪明”，而在**可追溯、可解释、可复现**：任何一版方案都必须知道哪些参数来自用户、哪些来自系统推断、哪些来自规则补全、对应的标准库/规则版本是什么，否则 BOM 与 PDF 都不可审计。eBOM 本身就强调设计变更过程中的持续更新与版本关联，这个系统也必须具备同样的“版本可追溯性”。citeturn22view0turn22view2

首版 MVP 的边界建议如下：

| 维度 | 首版建议 |
|---|---|
| 结构范围 | 正交矩形置物架、储物架、展示架、带脚轮推车 |
| 拓扑范围 | 立柱、横梁、纵梁、层框、层板支撑、中横梁、背撑、脚杯、脚轮、常见连接件 |
| 尺寸输入 | 长宽高、层数、层高、单层承重、是否带脚轮 |
| 交互方式 | 自然语言为主，点选/结构树/属性面板辅助 |
| 手工能力 | 仅允许在 AI 结果之上做有限辅助编辑 |
| 检测能力 | 经验规则 + 基础合理性检查，不做工程仿真 |
| 导出能力 | 3D 预览、结构树、设计 BOM、交易 BOM、尺寸标注视图、PDF |
| 不做内容 | 自由建模、复杂工况、实时价格、直接下单、实时协作、公开分享链接 |

能力拆分建议是：**首版做“约束内成功率”，后续版本做“结构类型扩展”**。最优顺序不是先把 AI 做强，而是先把**结构域、标准库和规则边界做窄**。因此建议后续版本按下面的方向扩展，而不是首版一起上：

| 阶段 | 应新增的能力 |
|---|---|
| 首版之后 | 门型架、带背板/侧板、抽屉框、带门、可选层板材质 |
| 中期 | 光轴、线性导轨、第三方配件、非矩形子模块 |
| 后期 | 账号体系、链接分享、订单打通、价格系统、供应链库存策略 |
| 最后 | 多用户协作、企业权限、复杂工况分析、私有化配置中心 |

### 产品交互与页面架构

产品交互必须是一条很短的主链路：**输入 → 确认 → 生成 → 编辑 → 检测 → 导出**。其中“确认”不是礼貌性步骤，而是系统正确率的核心保障：关键参数缺失时先问；非关键参数允许补默认值；生成前明确标出“系统推断值”；如果结构超出首版支持边界，在确认层就拦住，而不是等 3D 失败后再解释。由于你要面向非 CAD 用户，交互上不应该把用户推进一个多页向导，而应采用**单工作区、分阶段状态切换**的方式，让用户始终在同一上下文里看见“需求、草案、结果、修改”。citeturn20view4turn28search5turn28search2

完整用户流程建议这样设计：

1. **输入需求**：聊天输入框接受自然语言；旁边给示例 prompt；缺失关键参数时即时追问。  
2. **确认层**：显示 AI 理解出的结构类型、尺寸、层数、承重、是否脚轮、推断值、候选模板与不支持项。  
3. **生成阶段**：点击确认后进入前端本地建模，先生成“装配草案”而非最终文档。  
4. **编辑阶段**：用户可点选对象，结构树与属性面板联动；自然语言编辑默认以“当前选中对象”为上下文。  
5. **检测阶段**：用户主动点“检测”后，系统跑基础规则并给建议；检测结果支持一键应用或忽略。  
6. **导出阶段**：导出 3D 方案快照、设计 BOM、交易 BOM、尺寸标注视图和 PDF。  

页面信息架构建议采用**单页工作区**而不是传统应用多页跳转。推荐布局如下：

| 区域 | 主要职责 |
|---|---|
| 顶部栏 | 新建、撤销重做、快照、检测、导出 |
| 左侧栏 | 聊天输入、AI 理解、确认层、编辑建议 |
| 中央区 | 3D 场景、点选、视角切换、尺寸视图切换 |
| 右侧栏 | 结构树 / 属性面板 / BOM / 检测结果 选项卡 |
| 底部区 | 命令历史、版本快照、问题提示 |

局部编辑交互要坚持一个原则：**自然语言是主入口，3D 点选只是消歧器**。技术上，Babylon.js 的 PointerDragBehavior、BoundingBoxGizmo、Selection Outline 和 Utility Layer 很适合做“选中—高亮—局部拖拽/调整—覆盖层操作”的编辑器交互；如果切到 Three.js，则需要 Raycaster、TransformControls、OutlinePass 及额外的标签渲染能力来拼装同样的体验。尺寸与标注方面，Babylon GUI 基于 DynamicTexture，适合做工作区内标注覆盖层；Three.js 的 CSS2DRenderer 适合 HTML 标签，但其缩放限制不适合把它当成首版的主尺寸标注方案。citeturn28search0turn28search1turn28search2turn27search3turn27search4turn26view1turn26view0turn26view2turn18view1

必须刻意避免把体验做成传统 CAD，至少有五点：**不要以草图/拉伸/约束求解为主心智；不要把坐标和矩阵暴露给用户；不要让用户手工逐件搭建；不要把属性面板做成 50 个字段表单；不要把结构树做成 feature history 树**。正确做法是：对用户显示“层、立柱、横梁、脚轮、加固”这些语义对象，对系统内部再映射到 profile/joint/fastener/mount 这些工程对象。

## 系统与数据建模蓝图

### 系统架构设计

推荐架构是**前端本地建模为主、后端能力封装为辅、规则内核前后端共享**。原因很简单：你的主要编辑操作是确定性的参数变更，不需要每次都回后端；而 AI 调用、规则版本管理、PDF 导出、供应商目录同步这些又天然适合后端。浏览器侧使用 IndexedDB 持久化本地数据、Web Worker 执行重算与校验、必要时再用 OffscreenCanvas/场景截图做快照，是符合现代 Web 能力边界的；MDN 也明确说明 IndexedDB 适合大量结构化数据且可在 Worker 中使用，Web Workers 可以在后台线程执行脚本而不阻塞 UI，OffscreenCanvas 可以在 Worker 中执行渲染任务。citeturn19view7turn20view1turn20view2turn20view0turn20view3

整体系统架构建议如下：

```text
[Browser SPA]
  ├─ Chat & Confirm UI
  ├─ Editor Domain
  │   ├─ Project Store
  │   ├─ Selection Store
  │   ├─ Command Journal
  │   └─ Snapshot Manager
  ├─ Rule Worker
  │   ├─ Schema Validation
  │   ├─ Defaults & Normalization
  │   ├─ Topology Synthesis
  │   ├─ BOM Generator
  │   └─ Detection Engine
  ├─ Parametric Modeler
  ├─ Babylon Renderer
  ├─ Dimension View Generator
  └─ IndexedDB Persistence

[API Layer]
  ├─ AI Orchestrator
  ├─ Standard Library / Rule Version Service
  ├─ Supplier Catalog Mapping Service
  ├─ Export Service
  └─ Telemetry / Eval Service

[Shared Packages]
  ├─ schemas
  ├─ stdlib
  ├─ rules
  ├─ modeler-kernel
  ├─ bom-kernel
  └─ export-contracts
```

最关键的架构约束只有两条。第一，**前后端共享同一套 schema 与规则包**，不要分别实现两套“差不多”的规则，否则 BOM 和检测会漂移。第二，**3D 场景永远不是 source of truth**，真实数据源只能是项目文档和 resolved DSL，3D 只是投影。这样做，才能同时保证快照、撤销/重做、检测、BOM、导出和重放的一致性。这个思路也与 eBOM/mBOM 的“设计结构是主源，制造/采购视图从其派生”一致。citeturn22view0turn22view2

前端与后端职责建议明确切开：

| 模块 | 放前端 | 放后端 |
|---|---|---|
| 3D 场景渲染 | 是 | 否 |
| 点选与局部编辑 | 是 | 否 |
| 参数化建模 | 是 | 否 |
| 快照/撤销/重做 | 是 | 否 |
| AI NLU 与意图解析 | 否 | 是 |
| 规则库版本分发 | 否 | 是 |
| 供应商目录 ETL | 否 | 是 |
| 设计 BOM 生成 | 是 | 可复算 |
| 交易 BOM 映射 | 可预览 | 是主真相 |
| PDF 方案单 | 否 | 是 |

这个项目很适合“前端本地建模为主”，但并不适合“纯前端闭环”。适合的部分是**低延迟编辑、结构树联动、局部重算、即时预览**；不适合的部分是**LLM 编排、目录数据同步、正式导出、审计与可追溯性**。因此最优形态不是“全前端”，而是**local-first editor + server-side intelligence and export**。

### 3D 建模与数据结构设计

数据结构必须采用**装配树与渲染树分离**。在 Babylon.js 里，TransformNode 是一种**不参与渲染但可作为变换中心的节点**，官方文档明确提到它还可以减少内存占用并提升渲染效率；这非常适合作为 assembly anchor。每个业务对象都应有稳定的 `id`，渲染层再把这个 `id` 写入 mesh metadata，用于拾取回溯、局部更新和 AI 引用解析。citeturn28search3turn28search6turn17search1

推荐的项目文档主结构如下：

```ts
type ObjectId = string;
type PartSpecId = string;
type JointId = string;

interface ProjectDocument {
  projectId: string;
  schemaVersion: string;
  stdlibVersion: string;
  ruleVersion: string;
  createdAt: string;
  updatedAt: string;
  rootAssemblyId: ObjectId;
  objects: Record<ObjectId, StructuralNode>;
  joints: Record<JointId, JointNode>;
  views: ViewState;
  commandLog: CommandEntry[];
  snapshots: SnapshotMeta[];
  aiContext: AiContextState;
}

type StructuralKind =
  | "assembly"
  | "profile"
  | "connector"
  | "shelf"
  | "brace"
  | "foot"
  | "caster"
  | "panel";

type StructuralRole =
  | "upright"
  | "beam_x"
  | "beam_y"
  | "shelf_frame"
  | "shelf_support"
  | "mid_beam"
  | "back_brace"
  | "side_brace"
  | "foot"
  | "caster"
  | "corner_joint";

interface StructuralNode {
  id: ObjectId;
  kind: StructuralKind;
  role: StructuralRole;
  parentId: ObjectId | null;
  children: ObjectId[];
  name: string;
  aliases: string[];            // 如 rear_left_upright / top_shelf_front_beam
  paramExpr: Record<string, ExprOrValue>;
  resolved: ResolvedGeometry;
  constraints: ConstraintRef[];
  renderRefs: RenderRef[];
  bomRef?: BomSeedRef;
  tags: string[];
}

interface ProfileNode extends StructuralNode {
  kind: "profile";
  specId: PartSpecId;
  axis: "x" | "y" | "z";
  start: Vec3;
  end: Vec3;
  cutLengthMm: number;
  machiningOps?: MachiningOp[];
}

interface JointNode {
  id: JointId;
  type: "angle_bracket" | "inner_fastener" | "gusset" | "join_plate";
  connects: ObjectId[];
  specId: PartSpecId;
  mountRules: string[];
}
```

AI、规则与建模之间不要只用一个大 JSON。最佳做法是**三段式 DSL**：  
**意图 DSL**描述用户想要什么；**草案 DSL**描述结构拓扑候选；**已解算 DSL**描述准确的型材、连接件、长度、位置和 BOM seed。这样能把“自然语言的不确定性”与“建模的确定性”分开。采用支持 JSON Schema 的结构化输出接口后，模型可以稳定返回符合 schema 的对象，但官方文档同样强调：JSON mode 只保证是合法 JSON，不保证符合你的业务 schema；要么使用 Structured Outputs，要么仍需在代码中做校验。citeturn18view3turn18view4turn29view0turn29view1

建议的 DSL 分层如下：

```json
{
  "intentDsl": {
    "archetype": "storage_rack",
    "dimensionsMm": { "length": 1200, "width": 600, "height": 1500 },
    "tiers": [
      { "index": 1, "loadKg": 40, "purpose": "material_box" },
      { "index": 2, "loadKg": 25, "purpose": "printer" }
    ],
    "mobility": { "hasCaster": false },
    "reinforcementIntent": ["more_cross_beams"],
    "inferred": [
      { "path": "profile.defaultSeries", "value": "3030", "reason": "span_and_load_guess" }
    ],
    "needsConfirmation": []
  },
  "draftDsl": {
    "templateCandidate": "rack_two_tier_basic",
    "topology": [
      "4 uprights",
      "2 shelf frames",
      "1 optional mid beam per tier",
      "1 optional back brace"
    ]
  },
  "resolvedDsl": {
    "nodes": [],
    "joints": [],
    "checks": [],
    "designBomSeed": []
  }
}
```

局部编辑的难点不是“改长度”，而是**如何把“这根”“这一层”“后面那根”正确绑定到对象**。这里建议做一个**对象引用解析层**，其输入不是原始聊天文本，而是“聊天文本 + 当前选中对象 + 当前悬停对象 + 当前结构树焦点 + 当前相机朝向 + 最近一次解析到的对象锚点”。本质上，局部编辑不是单纯 NLP，而是**多模态指代消解**。citeturn26view1turn28search1turn17search1

推荐的数据接口如下：

```ts
interface ReferenceContext {
  selectedIds: ObjectId[];
  hoveredId?: ObjectId;
  treeFocusId?: ObjectId;
  cameraDirection: "front" | "rear" | "left" | "right" | "top" | "iso";
  visibleAliases: string[];
  recentAnchors: ObjectId[];
}

interface ReferenceQuery {
  pronoun?: "this" | "that" | "here";
  roleHint?: StructuralRole;
  tierHint?: "top" | "bottom" | "middle" | number;
  spatialHint?: "front" | "rear" | "left" | "right" | "center";
  relationHint?: "same_tier" | "behind_selected" | "next_to_selected";
}
```

解析顺序要固定：**显式选中 > 结构树焦点 > 最近锚点 > 视角相对位置 > 角色查询 > 追问澄清**。这样“把这根加长”在有选中对象时绝不需要再次做大范围语义猜测，而“把后面那根加长”才会调用视角与结构关系解析。

尺寸标注视图不能直接依赖渲染层的可见包围盒来临时生成，而要使用**解算后的装配几何**来组织。Three.js 的 Box3 很适合 bound 计算；Babylon.js 也提供 BoundingInfo、Lines 与 Line System / Dashed Lines，以及 GUI 覆盖层能力。更可靠的做法是：对每个视图预先生成一个 `DimensionViewModel`，其中定义**测量对象、测量线、延长线、箭头、标签、单位和隐藏策略**，导出 PDF 时直接复用这一份数据，而不是重新从场景里测一遍。citeturn26view6turn27search5turn27search1turn27search4turn16search2

再明确一次选型结论。Three.js 的强项在于：基础对象模型清晰，Raycaster、TransformControls、InstancedMesh、Box3、GLTFLoader / Exporter 都很好用；如果团队已经长期深耕 Three.js，也能做出高质量编辑器。Babylon.js 的强项在于：**更像一个“3D 应用框架”而不是“渲染内核”**，首版编辑器所需的大部分高层能力都能直接拿到。综合你的团队配置和首版目标，应该优先把研发预算花在**规则正确性与交互闭环**上，而不是花在 3D 基础设施拼装上，所以结论仍然是 **Babylon.js 优先**。citeturn26view0turn26view1turn26view2turn26view5turn26view6turn26view3turn28search3turn28search1turn28search2turn27search3turn2search3

## AI 与规则执行蓝图

### AI 能力链路设计

AI 的职责应被严格限制为四件事：**需求理解、草案生成、局部编辑意图生成、检测解释**。除此之外，包括参数合法性、型材与连接件兼容性、长度解算、默认值补全、BOM 生成、风险判定，都不应该由 AI 最终裁决。最佳链路不是“一次大模型调用生成全部结果”，而是**多步、短上下文、强 schema 的流水线**。支持 Structured Outputs 的接口正适合这件事：模型输出可以直接受 JSON Schema 约束；如果要用 function calling，也应把函数定义成小而清晰的 JSON Schema 工具，而不是让模型自由吐一大段 JSON。citeturn18view3turn18view4turn29view0turn29view1turn29view2

推荐把自然语言理解拆成五个子任务：

| 子任务 | 输入 | 输出 |
|---|---|---|
| 需求解析 | 用户原始文本 | intentDsl |
| 缺参判断 | intentDsl + 关键参数规则 | clarification list |
| 模板检索 | intentDsl | template candidate |
| 草案生成 | intentDsl + template/library summary | draftDsl |
| 局部编辑解析 | 编辑文本 + referenceContext + current model summary | editIntentDsl |

建议的函数工具集合如下：

```json
[
  {
    "name": "extract_requirements",
    "purpose": "从自然语言中抽取结构需求与关键参数",
    "output": "intentDsl"
  },
  {
    "name": "resolve_missing_params",
    "purpose": "判断哪些参数必须追问，哪些可以补默认值",
    "output": "confirmationDsl"
  },
  {
    "name": "propose_structure_draft",
    "purpose": "根据模板候选或规则候选生成结构草案",
    "output": "draftDsl"
  },
  {
    "name": "resolve_local_edit_intent",
    "purpose": "在 referenceContext 下解析局部编辑对象与动作",
    "output": "editIntentDsl"
  },
  {
    "name": "explain_check_results",
    "purpose": "把检测结果翻译成用户可理解说明",
    "output": "checkExplanationDsl"
  }
]
```

确认层建议独立建模为 `confirmationDsl`，其价值不只是展示，而是充当**生成前的契约**。建议字段至少包括：`userProvided`、`inferred`、`defaulted`、`missingCritical`、`unsupportedSemantics`、`requiresConfirmation`、`templateFallbackCandidate`。这样系统才能明确告诉用户：“长度宽度高度来自你的原话；型材规格由系统推断；脚轮缺失所以已追问；你提到的防震环境不在首版范围。”这种“参数补全 + 推断值标记”能显著降低后续争议。citeturn18view3turn29view0turn29view1

减少 AI 幻觉的核心不是“多写 prompt”，而是**收缩输出空间**。建议同时做八件事：  
其一，AI 侧只能看到**压缩后的标准库摘要**，不能自由发明零件名。  
其二，所有可选 profile / joint / mount 类型都走 enum，不给自由文本。  
其三，AI 只输出候选结构与编辑意图，绝不直接输出最终 BOM。  
其四，规则引擎对 AI 输出每一步都做 schema 校验、默认值补全与合法化。  
其五，局部编辑必须带 `referenceContext`，没有上下文就追问。  
其六，检测解释必须基于**规则引擎输出的 issue 列表**生成，不允许 AI 自行构造风险。  
其七，如果使用结构化输出的 function calling，建议设置 `strict: true`；官方还特别说明当 Structured Outputs 与并行函数调用混用时可能不匹配 schema，因此应显式关闭并行工具调用。  
其八，生成失败后只允许回退到“接近模板推荐”，不允许 AI 自行换一个超范围结构。citeturn29view1turn18view3turn18view4turn29view0

必须明确哪些事情**不能**交给 AI：  
不能交给 AI 的包括**长度计算、连接件兼容性判断、装配约束、配件数量结算、BOM 汇总、SKU 映射、检测结论**。这些内容都属于系统真相层，必须是确定性的。AI 最多只能说“建议加背撑”，但到底是否需要、加在哪、用什么规格、是否与当前 line 兼容，都必须由规则层裁决。

### 规则引擎与检测模块设计

这个项目不适合一上来引入“通用规则引擎框架”来解决全部问题，最优方案是**配置化规则目录 + TypeScript 算法核**。原因在于：很多规则并不是简单的 `if-then`，而是涉及**结构拓扑补全、长度推导、连接件生成、向量/坐标计算、受影响节点传播**。这些内容更适合用可测试的纯函数写在 TS 内核中；而阈值、兼容矩阵、推荐文案、优先级、适用范围，则适合做成 JSON/YAML 配置。citeturn22view4turn24view0turn24view1turn24view2

规则建议分成五层：

| 规则层 | 主要职责 |
|---|---|
| schema 校验层 | 校验字段完整性、类型、范围、禁用组合 |
| 归一化层 | 自动补默认值、单位统一、语义转枚举 |
| 拓扑生成层 | 立柱/横梁/层框/加固/底部构件的自动补足 |
| 兼容与合法化层 | profile line、连接方式、安装方式、配件搭配 |
| 检测与建议层 | 稳定性、跨度、承重、脚轮/脚杯、加固建议 |

检测模块必须做成**独立纯函数模块**，而不是嵌在生成流程里。你的产品需求已经明确“默认不主动触发建议”，这从架构上也成立：检测是**对当前 resolved model 的分析**，它不应和 AI 生成过程耦合。建议接口如下：

```ts
interface CheckInput {
  project: ProjectDocument;
  resolvedDsl: ResolvedAssembly;
  loadSpec: LoadSpec;
  stdlibVersion: string;
  ruleVersion: string;
}

interface CheckIssue {
  code: string;
  severity: "info" | "warn" | "error";
  scope: { nodeIds: string[]; jointIds?: string[] };
  title: string;
  detail: string;
  evidence: Record<string, unknown>;
  suggestions: CheckSuggestion[];
}

interface CheckSuggestion {
  action: "add_mid_beam" | "upgrade_profile" | "add_back_brace" | "switch_mount" | "adjust_tier_height";
  targetScope: string[];
  previewPatch?: object;
}
```

首版可做的基础检测，建议只覆盖**能被经验规则稳定描述**的一类问题，而不是假装在做工程分析。公开的型材系统资料和工具已经说明，行业常用判断维度主要还是**型材截面、跨度、载荷、支撑条件、连接加固、底部构件**。因此首版的检测范围建议是：  
**是否缺少关键参数**、**是否出现不兼容的型材/连接件组合**、**单层跨度与选定型材规格明显不匹配**、**高瘦结构缺少背撑/侧撑**、**重载层缺少中横梁**、**脚轮方案与结构重心/底部结构不协调**、**底部缺少稳定支撑构件**、**连接方案在当前 line 下不成立**。45° 支撑件官方就明确被用于加固 90° 连接并减小重载场景下的梁挠度；脚轮和脚杯目录也明确体现了承载能力、安装方式与地面稳定性差异，所以这些都适合作为显式规则项。citeturn23view0turn18view7turn23view3turn23view2turn23view1turn22view5

首版不应该做的内容同样要明确：**环境区分、地震/冲击载荷、多材料耦合、连接件受力仿真、螺栓预紧、真实地面不平整建模、疲劳寿命、复杂跌落与侧翻仿真**。这不是说以后永远不做，而是这些能力一旦引入，就会把系统从“DIY 装配设计器”拖向“工程 CAE 工具”，完全超出 MVP。

未来如果要扩展到光轴、第三方配件，规则体系不要重写，而应增加**部件族适配层**。也就是所有可扩展零件都先挂到统一接口：`partFamily`、`mountPattern`、`constraintGroup`、`kinematicRole`、`bomCategory`。规则引擎只认识这些抽象接口，不直接认识某个供应商的商品页。

## 标准库与工程实现蓝图

### 标准库、BOM、SKU 映射设计

标准库最重要的设计思想只有一句话：**设计阶段认识“规格”，交易阶段认识“SKU”**。这是本项目能否长期健康演化的分水岭。因为公开供应商资料已经非常清楚地显示，型材系统同时存在**不同 modular line、不同 second line、不同连接方式、不同安装条件、同一连接族的多个材质/表面处理/承载版本**；如果一开始就把设计对象绑定成供应商 SKU，后续替代、缺货、第三方扩展都会痛苦。citeturn24view0turn24view1turn24view2

抽象标准库建议至少包含以下对象：

| 对象 | 必备字段 |
|---|---|
| ProfileSpec | family、series、line、outerSize、slotType、unitWeight、sectionRefs、compatibleJoints |
| JointSpec | jointFamily、angleType、compatibleLines、requiresMachining、includedHardware、strengthClass |
| FloorSpec | foot/caster/basePlate、mountPattern、loadClass、requiresBasePlate、mobilityClass |
| HardwarePack | screw、tNut、washer、nut、countRule |
| PanelSpec | material、thickness、mountMode、future use |
| SupplierSku | supplierPartNo、specRef、packQty、minOrderQty、status、substituteGroup |
| SubstituteGroup | preferredSku、fallbackRules、equivalenceClass |

设计 BOM 应按 eBOM 思路组织，表达“**按设计的结构**”：它需要保留层级、角色、位置、长度、规格、连接关系、是否由系统补全等信息。交易 BOM 则更接近 mBOM / 采购视图，表达“**实际要买什么**”：它需要把多个设计对象汇总到同一采购项，合并紧固件包，处理包装数量、替代件和缺货规则。公开 BOM 资料本来就是把 eBOM 定义为 as-designed 结构，把 mBOM 定义为 as-planned / as-built 的制造视图；你的“设计 BOM / 交易 BOM”正适合沿这条成熟思路落地。citeturn22view0turn22view1turn22view2

建议的双 BOM 结构如下：

```ts
interface DesignBomItem {
  bomId: string;
  objectId: string;
  role: string;
  specId: string;
  description: string;
  quantity: number;
  unit: "pcs" | "mm" | "set";
  geometry?: { cutLengthMm?: number };
  parentBomId?: string;
  source: "user" | "default" | "rule";
}

interface TradeBomItem {
  tradeBomId: string;
  designRefs: string[];
  procurementSpecId: string;
  skuCandidates: string[];
  selectedSku?: string;
  quantity: number;
  packRule?: { minOrderQty: number; packQty: number };
  substituteGroup?: string;
  supplierHints?: string[];
}
```

规格到 SKU 的映射不要做成单一查表，而要做成**多阶段决策**：  
第一阶段，按 `specId` 找所有满足规格的 `skuCandidates`。  
第二阶段，按供应商优先级、包装规则、缺货状态、交期、替代等级筛选。  
第三阶段，生成 `selectedSku` 与 `fallbackSkuList`。  
第四阶段，把“同规格多对象”合并成交易行，并补足包装尾数。  

如果接入的是类似目录型供应商站点，而不是标准 API，建议后端先做**离线 ETL 标准化**，再把规范后的 catalog feed 提供给前端与规则层，绝对不要在运行时直接拿商品页 HTML 做规则判断。公开目录也显示出很多产品同时带有 line、second line、连接类型、分类码（如 eCl@ss / UNSPSC）和 ERP/e-business 数据，这类字段非常适合作为你内部 catalog normalization 的中间层。citeturn24view2turn24view1

缺货替代和 “一个规格对应多个 SKU” 在这个项目里不是边缘问题，而是原则问题。建议从第一天就引入三个字段：`equivalenceClass`、`substitutePriority`、`supplierPolicy`。这样交易层才能处理这些典型场景：  
其一，同规格不同表面处理。  
其二，同规格不同供应商。  
其三，同一连接族在不同材质下可互替。  
其四，重载脚轮需要 base plate，而轻载脚轮可直接装。  

公开型材目录里脚轮、脚杯、底板、连接件本来就体现出这种“规格兼容但安装方式不同”的关系，因此替代逻辑必须是**规则映射**，不能只靠关键词搜索。citeturn18view6turn23view3turn23view2turn24view1

主导出 3D 格式建议使用 **GLB / glTF + project JSON**。原因很直接：glTF 是开放的运行时 3D 传输格式，Khronos 规范和 Babylon / Three 官方都提供 glTF 级别的导出与加载能力；而 project JSON 则保留你的业务语义与可编辑性。也就是说：**GLB 给外部预览，project JSON 给内部重编辑**。citeturn13search0turn26view3turn26view4turn2search3

### 前端工程与开发方案

推荐采用 **Vue 3 + TypeScript + Vite + Pinia + Babylon.js** 的单仓/多包结构。Vue 3 的 Composition API 本身就是为复杂逻辑组合而设计，官方也明确把 `provide/inject` 作为 Composition API 的一部分；Pinia 官方强调其 type-safe、extensible、modular，非常适合把工作区按 bounded context 拆成多个 store，而不是一个巨型全局状态。citeturn21view2turn21view1turn21view0

工程目录建议按“领域内核优先”组织，而不是按页面组织：

| 包名 | 职责 |
|---|---|
| `packages/domain` | 核心类型、命令、项目文档、选择模型 |
| `packages/schemas` | DSL schema、AI contracts、校验器 |
| `packages/stdlib` | 型材/连接件/目录规范化数据 |
| `packages/rules` | 合法化、检测、BOM 内核 |
| `packages/modeler` | 参数化装配到几何解算 |
| `packages/render-babylon` | Babylon 场景适配层 |
| `packages/export` | PDF/BOM 导出合同与模板 |
| `apps/web` | Vue 工作区应用 |
| `apps/api` | AI 编排、导出、目录服务 |

状态管理不要按“页面”拆，而要按“领域”拆。建议至少有六个 store：  
`projectStore`、`selectionStore`、`aiSessionStore`、`sceneStore`、`checkStore`、`exportStore`。  
其中 Babylon 场景对象不要直接放进 Pinia 响应式树里，而应通过 `markRaw` 服务对象 + store 中的轻量句柄关联，避免把场景节点变成巨大的响应式依赖网。Vue 的依赖注入正适合把这些 service 以 typed `InjectionKey` 方式注入给组件树。citeturn21view1turn21view2

编辑器状态与撤销/重做，建议采用**命令模式 + 增量 patch + 周期性快照**。原因是你的操作不是简单表单修改，而是“加一根梁”“把这一层抬高”“切换连接方式”“应用检测建议”这种会影响多个节点的复合动作。最佳实现不是每次都深拷贝整个 project，而是：  
- 每个用户操作生成一个 `Command`；  
- `Command` 输出 `patch` 与 `inversePatch`；  
- 每隔若干步生成一个完整 `snapshot`；  
- `snapshot` 与命令日志持久化到 IndexedDB；  
- 小量偏好与最近打开项目索引写入 localStorage。  

MDN 明确指出 localStorage 适合会话外持久化的小量字符串数据，而 IndexedDB 适合大量结构化数据、支持事务并能存储 structured clone 支持的对象。citeturn20view0turn19view7turn6search2turn6search6

3D 场景、结构树、属性面板必须彻底解耦。正确依赖方向应该是：  
**domain document → selectors → view models → 组件渲染**，  
而不是  
**组件 A 直接改 Babylon mesh，组件 B 再从 Babylon scene 读状态**。  
简单说：场景层只消费 domain，永远不拥有业务真相。

局部重算与局部更新的关键是建立**脏区传播图**。建议每次命令执行后，只标记受影响节点，然后按固定顺序重算：  
`topology dirty set → geometry dirty set → bom dirty set → render dirty set → dimension dirty set`。  
这能避免全量重建。对于渲染层，再应用一条很重要的优化策略：**使用混合渲染模型**。Babylon 的 thin instance 把位姿数据打包进矩阵，适合大量重复件；官方也说明 thin instances 的 picking 需要在矩阵列表变更后更新 picking list，而 Selection Outline 支持 instanced / thin-instanced meshes。结合这些约束，首版应把**处于当前编辑焦点外的大量重复连接件**渲染为 thin instances，把**当前可编辑范围内的对象**保持为普通 mesh 或普通 instance，以换取稳定点选和更简单的局部替换。citeturn27search7turn27search19turn27search3turn27search11

关于是否引入 worker、CRDT、命令模式、事件总线、状态机，建议如下：  
**worker：建议引入**，但首版只放规则重算、BOM 生成、检测与大对象序列化，不要一开始就把整个渲染搬进 worker。Web Workers 与 OffscreenCanvas 的官方能力足够支撑这一步，Vite 也原生支持 `?worker` 构建。  
**CRDT：不建议首版引入**，Yjs 官方明确定位是协作应用和自动合并并发更改；你首版没有账号、没有协作、没有多端同步，CRDT 只会增加复杂度。  
**命令模式：必须引入**，这是撤销/重做、快照、检测建议应用的基础。  
**事件总线：仅限域内强类型事件**，不要使用全局弱类型 `mitt` 式总线充当架构胶水。  
**状态机：建议引入**，但只用于“输入/确认/生成/检测/导出”这类工作流状态，而不是拿来管理整个场景。XState 官方文档强调其适合事件驱动、状态机和 actor 模型下的复杂逻辑，这正适合工作流，不适合替代 domain document。citeturn20view1turn20view2turn20view3turn20view5turn20view4

## 服务端与交付蓝图

### 后端与部署方案

后端的最小职责集应该非常克制，只做五件事：**AI 调用编排、标准库/规则版本分发、供应商目录标准化、正式导出、评测与日志**。除此之外，凡是能在前端本地同步完成的事情都不要抢到后端去做。这样你的 MVP 才能真正享受到“local-first editor”的低延迟价值。citeturn29view2turn18view2

AI 调用层建议独立成 `ai-orchestrator` 服务，内部只暴露少数几个 API：  
`/parse-requirements`、`/propose-draft`、`/resolve-edit-intent`、`/explain-checks`。  
每个 API 都应绑定固定 schema、固定 prompt 模板版本、固定工具列表，并记录 `promptVersion`、`stdlibVersion`、`ruleVersion`、`modelVersion`。如果采用支持 function calling 与 structured output 的接口，建议统一在服务端封装，并在工具定义层面保持 `strict: true` 与 `parallel_tool_calls: false` 的策略，避免 schema 漂移。官方文档已经明确指出 JSON mode 不等于 schema 保证，而 Structured Outputs 可以保证符合提供的 JSON Schema；同时，并行函数调用与严格结构化输出并不兼容。citeturn18view3turn18view4turn29view0turn29view1turn29view2

标准库与规则配置服务建议做成**版本化静态包 + 元数据索引**。最简单的 MVP 方案是：  
- 标准库 JSON、规则 JSON、模板 JSON 存对象存储 / 静态文件服务；  
- API 只返回当前发布版本、变更说明和文件索引；  
- 前端缓存到 IndexedDB；  
- 项目文档保存所依赖的版本号。  

这样既保留了前端本地运行能力，也确保了“今天生成的 BOM、明天还能按同一版本复现”。

PDF 导出建议**以后端为主**。原因不是前端不能导，而是要追求**排版稳定、字体可控、页眉页脚一致、尺寸视图复现一致**。Playwright 官方的 `page.pdf()` 直接基于页面生成 PDF，并支持 print CSS、页面尺寸、边距与媒体模式，这非常适合把 HTML 方案单模板稳定地导成 PDF。前端可以保留一个“快速浏览器打印”的降级路径，但正式导出必须以后端 HTML → PDF 为准。citeturn18view2

MVP 部署建议采用一套很轻的架构：

```text
CDN / Static Hosting
  └─ Web SPA

Node API
  ├─ AI Orchestrator
  ├─ Library/Rule Service
  ├─ Catalog Mapping Service
  └─ Export Service

Storage
  ├─ Object Storage for stdlib / templates / pdf
  └─ Relational DB for catalog normalization / logs / evals
```

如果做 SaaS，重点在**版本迭代快、目录同步频繁、AI 日志可观测**；如果做私有化，重点在**模型接口可替换、供应商目录可离线导入、导出服务可内网运行、规则库可热更新但必须留审计记录**。首版最优部署不是上来拆很多微服务，而是**单体 Node 服务 + 清晰模块边界**。你的团队体量很小，过早微服务化只会拖慢交付。

### 开发计划与里程碑

MVP 的最小可交付范围建议再压缩一次：**只要能把 4 类常见货架/置物架自然语言需求稳定落地，支持 5 类局部编辑动作，输出设计 BOM / 交易 BOM / PDF，就可以上线闭环验证**。不要试图在首版就覆盖“任意结构”或“所有型材体系”。

建议按下面的阶段推进：

| 阶段 | 核心目标 | 关键产出 | 必须程度 |
|---|---|---|---|
| 域模型阶段 | 建立项目文档、标准库、规则骨架 | schemas、stdlib、fixtures | 必须先做 |
| 建模阶段 | 前端参数化建模与 3D 工作区 | 结构树、点选、渲染、局部重算 | 必须先做 |
| 规则阶段 | 合法化、BOM、检测 | resolvedDsl、design BOM、checks | 必须先做 |
| AI 初始生成阶段 | 自然语言转 intent/draft | 确认层、模板回退 | 必须先做 |
| AI 局部编辑阶段 | 选中对象上的自然语言编辑 | editIntent、局部 patch | 必须先做 |
| 导出阶段 | PDF 与采购清单 | HTML 模板、PDF、CSV/Excel | 必须先做 |
| 体验打磨阶段 | 快照、撤销、稳定性 | snapshots、错误恢复 | 可后补但尽早 |
| 供应链增强阶段 | 替代件、缺货策略 | supplier policy | 可后补 |

“必须先做”和“可以后补”的分界线非常明确。**必须先做**的是：  
- 稳定的数据模型  
- 规则合法化  
- 基本 3D 编辑  
- 设计 BOM  
- 基础检测  
- 确认层  
- PDF 导出  

**可以后补**的是：  
- 多供应商策略  
- 高级模板推荐  
- 更多材质与面板  
- 门、抽屉、导轨  
- 协作、账号、分享  
- 价格、下单、库存

风险最高的模块排序建议如下：

| 风险等级 | 模块 | 原因 |
|---|---|---|
| 最高 | 局部编辑指代消解 | “这根/这一层/后面那根” 最容易错 |
| 很高 | 规则合法化与拓扑补全 | 直接决定是否可装配 |
| 很高 | 设计 BOM 到交易 BOM 映射 | 最容易和标准库耦死 |
| 中高 | PDF 与尺寸视图一致性 | 容易出现视觉与数据不一致 |
| 中高 | 检测建议可信度 | 过松无价值，过严会误伤 |
| 中等 | 首次自然语言生成 | 通过确认层可显著降风险 |

对于 **1~2 名前端 + 1 名后端/全栈 + AI 支持** 的团队，最现实的推进方式是：  
- **前端 A**：负责 domain document、规则 worker 接口、结构树、撤销/重做、工作区 UI。  
- **前端 B**：负责 Babylon 场景、点选、渲染、尺寸视图、局部更新。  
- **后端/全栈**：负责 AI 编排、标准库服务、目录规范化、导出与日志。  
- **AI 支持**：不是单独一个“AI 模块”，而是贯穿在 schema 设计、prompt 模板、评测样例和失败案例回放里。  

最重要的是节奏：**先把 domain、rules、modeler 跑通，再接 AI，而不是反过来**。

## 质量保障与路线建议

### 测试与验收

首版测试不能只做页面点点点，而必须把测试重心压在**规则正确性、模型可复现性和 AI 合同稳定性**上。推荐测试金字塔如下：  
底层用 **Vitest** 跑 domain、rules、BOM、resolver 的纯函数测试；  
中层用组件与浏览器测试验证结构树、属性面板、确认层交互；  
顶层用 **Playwright** 跑完整 E2E，包括从输入需求到导出 PDF 的主链路。Vitest 官方支持浏览器模式，Playwright 官方支持截图比较与可视回归，这两者正适合你的场景。citeturn21view3turn21view4turn21view5

关键验收用例建议至少覆盖这些场景：

1. 明确参数输入，能直接生成双层置物架。  
2. 缺失单层承重，系统能追问而不直接瞎猜。  
3. 参数不完整但非关键，确认层能标出推断值。  
4. 选中一根梁后说“把这根加长到 900”，只影响局部。  
5. 选中一层后说“这一层抬高 100”，相关梁和立柱同步调整。  
6. 说“这里加一根横梁”，系统能在正确层位补足构件与连接件。  
7. 检测能提示缺少背撑 / 中横梁 / 型材升级建议。  
8. 设计 BOM 与结构对象数量完全对应。  
9. 交易 BOM 能把同规格对象正确汇总到采购项。  
10. PDF 中的尺寸视图、BOM、文本摘要与当前模型一致。  
11. 快照恢复后，3D、结构树、检测结果和 BOM 都能复原。  
12. AI 输出非法结构时，规则引擎能拒绝并回退到确认或模板建议。  

评估指标建议这样设：

| 指标 | 建议目标 | 判定方式 |
|---|---|---|
| 关键参数抽取正确率 | ≥ 95% | 长宽高、承重、脚轮 on golden prompts |
| 首次生成成功率 | ≥ 85% | in-scope prompt 一次通过到 resolvedDsl |
| 局部编辑目标解析准确率 | 有选中时 ≥ 90% | objectId 命中率 |
| 局部编辑结果正确率 | ≥ 85% | patch 后结构与预期比对 |
| 设计 BOM 正确率 | ≥ 99% | golden BOM exact match |
| 交易 BOM 映射覆盖率 | ≥ 95% | 目录可用规格集合上的映射覆盖 |
| 检测建议有效率 | ≥ 70% | 由人工评审判定“建议合理” |
| PDF 一致性 | 100% 关键项一致 | 视图快照 + 表格内容校验 |

回归测试与样例集必须长期维护，并且按四类样本组织：  
**需求样本集**用于测初始生成；  
**编辑样本集**用于测局部修改；  
**BOM 样本集**用于测规格与数量；  
**检测样本集**用于测 issue 与建议。  
另外再补一类**失败样本集**：所有线上失败 case 都要能回放。这样 AI 与规则升级时，才能知道自己到底修好了什么、又破坏了什么。

视觉回归测试建议只盯三个东西：**确认层、结构树状态、尺寸视图**。Playwright 官方已给出截图对比能力，但也明确提醒操作系统、硬件与渲染环境会影响截图，因此基线与 CI 环境必须固定。citeturn21view5

### 风险与建议

这个项目最容易失败的地方，不是 3D 引擎，也不是模型调用，而是**产品边界失控**。一旦你允许“任意建模”“任意连接件”“任意第三方件”“任意复杂编辑”，AI、规则、建模、BOM、导出会同时失真，最后谁都不可信。真正的首版策略不是追求“覆盖面”，而是追求**在明确边界内非常稳定**。

技术债最高的点有六个：  
第一，把 Babylon/Three 场景树当作真相源。  
第二，把商城 SKU 直接写进设计模型。  
第三，把规则写进 prompt，而不是规则引擎。  
第四，没有稳定对象 ID，导致局部编辑无法可靠重放。  
第五，没有版本化标准库与规则包，导致历史方案不可复现。  
第六，把 AI 生成、规则合法化和检测结果混成一个过程，失去可审计性。

不应过早投入的方向也很明确：**复杂仿真、实时价格、购物车打通、账号体系、多人协作、自由建模、第三方全量接入**。其中多人协作尤其要谨慎；Yjs 之类 CRDT 方案很强，但它服务的是并发修改与自动合并，而你首版没有账号和协作场景，提前引入只会把数据模型复杂度翻倍。citeturn20view5

最优首版策略建议可以浓缩为一句话：**把产品做成“会话式货架配置器”，而不是“浏览器里的 CAD”**。工程上应坚持五个原则：  
- **受约束生成**，不要自由生成。  
- **规则优先**，不要 AI 终裁。  
- **本地建模**，不要关键编辑依赖后端往返。  
- **规格先于 SKU**，不要设计期绑定商城。  
- **导出稳定**，不要临时拼接 PDF。  

后二期最值得扩展的方向，不是更多 prompt，而是更多**部件族**：光轴、导轨、门、抽屉、面板、护栏、第三方安装件。因为一旦首版把“抽象标准库 + 规则合法化 + 装配 DSL + 双 BOM”这条链路打通，新增能力主要就是补部件族与规则包，而不是推倒重来。

如果只保留一条最终建议，那就是现在该做的事情：**先冻结首版结构域，马上定义 DSL、标准库对象模型、规则层边界和 Babylon 工作区骨架；在这四件事完成之前，不要开始写大模型 prompt，也不要接商城 SKU。**
