# AI 驱动的 3D 铝型材网页设计系统首版工程执行蓝图

## 执行总览

### 1. 项目执行结论

1. 以 `resolvedDsl` 作为唯一真相源；`draftDsl` 只保留生成痕迹，3D、设计 BOM、交易 BOM、尺寸视图、PDF 全部从 `resolvedDsl` 派生。
2. 首版只做“矩形、正交、开放式、落地式”的置物架 / 储物架，不引入自由草图、非矩形框架、斜切、异形节点。
3. 仓库直接上 monorepo，前端、AI 合同、规则、标准库、BOM、建模、渲染、导出全部拆包；不要先写单体再拆。
4. AI 合同统一采用 entity["company","OpenAI","ai company"] Structured Outputs + function calling；所有字段默认必填，允许空值的字段显式用 `null` 表达，不接受纯 JSON mode 直出，因为官方明确区分了“合法 JSON”和“符合 schema”，而且 strict 模式只支持 JSON Schema 的一个子集。citeturn10view0turn10view1turn16view0turn16view3turn5search8
5. `confirmationDsl` 不交给 AI 最终裁定；它应由应用根据 `intentDsl` 和能力矩阵生成，再由用户确认，防止 AI 在歧义项上偷偷替你拍板。
6. `draftDsl` 只能表达“结构草案”和“拓扑意图”，不能出现最终 SKU、最终连接件型号、最终检测结论。
7. 规则引擎首版不要做“通用低代码规则平台”；直接做“typed rule registry + versioned config + TypeScript 纯函数”。
8. Pinia 只存文档、选择、命令、派生视图；Babylon Engine / Scene / Camera / Gizmo / AssetContainer / Mesh 等实例全部使用 `markRaw()` 或 `shallowRef()` 置于渲染适配层，禁止写入 ProjectDocument。Vue 官方把复杂第三方类实例明确列为不应深度响应式代理的对象，Pinia 则用于承载全局状态与业务逻辑。citeturn13view0turn13view1turn10view2turn10view3
9. 编辑命令必须是“语义命令”，不是“几何命令”。例如：`resizeBayWidth`、`insertShelfLevel`、`replaceProfileSeries`，而不是 `setNodeEndX`。
10. 渲染内核首版直接选 Babylon.js，不选 Three.js。原因不是抽象层面“谁更强”，而是 Babylon 已把首版刚需能力集中提供出来：utility layer、gizmo、selection outline、thin-instance picking、asset container、glTF loader；而 Three.js 同类能力在这个项目里更多依赖 addons 和额外集成。citeturn14search16turn9search4turn2search19turn2search28turn2search7turn9search1turn15view0turn15view1turn15view2
11. 规则层采用“全量重算业务结果 + 渲染层按差异局部更新”的策略。首版不要做业务级增量求解器，先保正确性。
12. 局部编辑必须走 `ReferenceContext -> editIntent -> target resolution -> semantic patch -> command` 这条链路，不允许 AI 直接改 `resolvedDsl`。
13. 本地持久化以 IndexedDB 为主，`localStorage` 只放最近项目、轻量 UI 偏好和索引；不要把完整项目文档塞进 `localStorage`。MDN 明确指出 IndexedDB 可事务化存储 structured-clone 支持的对象，而 `localStorage` 本质上是跨会话持久化的字符串键值存储。citeturn10view4turn10view5turn17view0turn17view1
14. 后端首版只做三件事：AI 代理、PDF 渲染、catalog ETL / catalog 下发；主要编辑动作不得依赖后端往返。
15. 第一轮验收不看“能不能画出来”，看五个硬指标：首次生成成功率、局部编辑正确率、设计 BOM 正确率、交易 BOM 映射正确率、检测建议有效率。

### 3. 仓库与包结构

`packages/schemas` 应以“手写、可审计、扁平化”的 JSON Schema 为准，不要以复杂 TS union 自动反推 schema。原因很直接：strict Structured Outputs 只支持 JSON Schema 的一个子集，schema 设计越花哨，越容易在上线时被 strict 校验反噬。citeturn16view0turn16view3

```text
repo/
  apps/
    web/
    api/
  tools/
    catalog-etl/
  packages/
    domain/
    schemas/
    stdlib/
    rules/
    bom/
    modeler/
    render-babylon/
    export/
    ai-contracts/
    test-fixtures/
```

| 包路径 | 角色 | 主要职责 | 依赖方向 | 是否必须 shared |
|---|---|---|---|---|
| `apps/web` | 主应用 | Vue 3 UI、Pinia、Babylon 集成、IndexedDB、交互层 | 依赖所有 packages | 否 |
| `apps/api` | 轻后端 | AI 代理、PDF 导出、catalog 下发、ETL 结果读取 | 依赖 `schemas/stdlib/rules/bom/export/ai-contracts` | 否 |
| `tools/catalog-etl` | 工具 | 原始 catalog CSV/XLSX -> 规范化 JSON | 依赖 `stdlib/bom/test-fixtures` | 否 |
| `packages/domain` | 核心域 | ProjectDocument、命令、引用、枚举、通用 DTO | 仅基础依赖 | 是 |
| `packages/schemas` | 契约层 | DSL JSON Schema、AI I/O Schema、runtime validators | 不依赖 UI / Babylon | 是 |
| `packages/stdlib` | 标准库 | 抽象标准库、规格级设计、兼容矩阵、SKU 层原始表 | 依赖 `domain` | 是 |
| `packages/rules` | 规则层 | 草案归一化、合法化、解析、检测 | 依赖 `domain/stdlib` | 是 |
| `packages/bom` | BOM 层 | 设计 BOM、交易 BOM、采购策略、SKU 映射 | 依赖 `domain/stdlib` | 是 |
| `packages/modeler` | 建模层 | `resolvedDsl -> param primitives / mesh plan` | 依赖 `domain/stdlib` | 是 |
| `packages/render-babylon` | 渲染层 | SceneAdapter、MeshPool、PickingMap、Highlighter | 依赖 `domain/modeler` | Web 专用 |
| `packages/export` | 导出层 | 尺寸视图 SVG、PDF payload、ZIP manifest | 依赖 `domain/bom` | 是 |
| `packages/ai-contracts` | AI 合同层 | prompts、tool schema、DTO、错误 envelope | 依赖 `schemas/domain` | 是 |
| `packages/test-fixtures` | 测试资产 | golden prompts、resolved fixtures、catalog fixtures | 可依赖所有 shared 包 | 是 |

**推荐依赖方向**

- `domain`、`schemas` 是底座，不能反向依赖应用层。
- `stdlib`、`rules`、`bom` 不能依赖 Babylon 或 Vue。
- `modeler` 只知道几何参数和视图模型，不知道 Scene。
- `render-babylon` 只消费 `modeler` 结果，不写业务真相。
- `apps/api` 不存项目真相；首版项目真相仍在浏览器本地。

**首版 API 面**

- `POST /v1/ai/intent`
- `POST /v1/ai/draft`
- `POST /v1/ai/edit-intent`
- `POST /v1/ai/check-explain`
- `GET /v1/catalog/:version`
- `POST /v1/export/pdf`

## 数据与规则

### 4. 核心数据模型

项目文档必须保持“plain object only”。MDN 说明 IndexedDB 可存储 structured-clone 支持的对象，`structuredClone()` 可深拷贝值并支持循环引用，但函数、DOM 节点等不可序列化；因此 ProjectDocument 里不应出现 Babylon 实例、函数、class 私有状态。citeturn10view4turn17view0turn17view1

**字段设计理由**

- `catalogVersion` 是首版必须字段。没有它，交易 BOM 无法复现，也无法解释“为什么今天和昨天映射结果不同”。
- `semanticPath` 是首版必须字段。没有它，局部编辑里的“这一层”“后面那根”“左侧那根立柱”无法稳定回指。
- `provenance.ruleIds` 是首版必须字段。否则检测解释和回放都很弱。
- `designBom`、`tradeBom`、`checkIssues` 建议作为 ProjectDocument 的“缓存派生物”持久化，但**理论上可重算**；这样能兼顾打开速度和迁移弹性。
- `excludeFromBom` 建议数据层预留、UI 二期化。因为它会影响设计 BOM、交易 BOM、导出、分组四个面，首版 UI 容易把范围带偏。
- `unitPrice`、`substituteSku`、`finishKey` 属于后续增强，首版不应用它们阻塞主链路。
- `CommandEntry.payload` 用统一 JSON 结构先跑起来，首版不必一开始就做成复杂 discriminated union；但 `type` 必须离散枚举，不能自由字符串。

### 5. DSL 与 AI 合同

首版 AI 合同必须极简。官方文档说明 Structured Outputs 可保证响应遵循提供的 JSON Schema；function tools 本质同样是 JSON Schema 驱动；纯 JSON mode 只保证输出是可解析 JSON，不保证符合指定 schema。官方还说明 strict 模式只支持 JSON Schema 的一个子集，并且 refusal 需要显式处理。所以首版策略是：schema 扁平、字段全必填、optional 改 nullable、所有 AI 输出先过本地 validator，再进入规则层。citeturn10view0turn10view1turn16view0turn16view1turn16view3turn5search8

**所有权冻结**

| DSL | 主要作用 | 谁产出 | 谁补全 / 修改 | 是否真相源 |
|---|---|---|---|---|
| `intentDsl` | 抽取用户需求、偏好与歧义 | AI | 应用可追加默认值标记 | 否 |
| `confirmationDsl` | 需要用户确认的决策清单 | 应用编译 + 用户确认 | 不由 AI 最终裁定 | 否 |
| `draftDsl` | 结构草案、拓扑意图、生成候选 | AI | 规则层可拒绝，不直接修改为真相 | 否 |
| `resolvedDsl` | 明确坐标、规格、节点、连接、派生依据 | 规则层 | 应用持久化 | 是 |

### 6. 规则引擎设计

**规则总原则**

- 首版不要发明“通用规则脚本语言”。
- 用 versioned config 管“表”，用 TypeScript 纯函数管“图、几何、推理、冲突处理”。
- 规则执行必须 deterministic，同一输入必须得到同一输出。
- 规则失败时必须产出 `CheckIssue`，而不是 silent fallback。

**规则分层**

| 层级 | 输入 | 输出 | 适合放配置 | 适合放 TS 纯函数 |
|---|---|---|---|---|
| 能力矩阵层 | `intentDsl / confirmationDsl` | 能否生成、哪些项需追问 | 是 | 否 |
| 草案归一化层 | `draftDsl` | `normalizedDraft` | 少量默认值 | 是 |
| 拓扑生成层 | `normalizedDraft` | `topologyPlan` | bay / level 模板 | 是 |
| 合法化层 | `topologyPlan + stdlib` | `legalizedPlan` | 阈值、允许范围 | 是 |
| 规格解析层 | `legalizedPlan + stdlib` | `resolvedDsl` | profile / connector 表 | 是 |
| 检测层 | `resolvedDsl + stdlib` | `CheckIssue[]` | 阈值、提示文案 key | 是 |
| 设计 BOM 层 | `resolvedDsl` | `DesignBomItem[]` | item recipe | 是 |
| 交易 BOM 层 | `DesignBomItem[] + skuMap` | `TradeBomItem[]` | supplier / SKU / pack policy | 是 |

**配置 vs TypeScript 边界**

适合配置的内容：

- profile family、spec 列表
- joint family 与可兼容矩阵
- 最大 span / 默认 shelf spacing / 最小边距
- BOM recipe
- SKU 映射表、stock bar 长度、pack size

必须写成 TypeScript 纯函数的内容：

- bay / level 展开
- 结构图生成
- 冲突消解和 tie-break
- 坐标求解
- 节点 / 连接拓扑推断
- 设计 BOM group key 计算
- 交易 BOM pack rounding
- 局部编辑 patch 规划

**首版必须实现的规则清单**

| ID | 类型 | 规则名 | 作用 | 单测 |
|---|---|---|---|---|

- 局部编辑不是绕过规则层的捷径；它只是另一种产生命令的入口。

## 客户端编辑器与 3D 内核

### 7. 前端编辑器设计

Vue 官方明确建议对复杂第三方类实例使用 `markRaw()`，并以 `shallowRef()` 承载大型外部状态；Pinia 则用于全局状态和业务逻辑。Babylon 侧又提供了 `onPointerObservable`、selection outline、utility layer、gizmo 和 thin-instance picking。基于这些能力，前端应采用“Pinia 管业务文档 + raw scene adapter 管 Babylon 对象 + ProjectDocument 驱动渲染”的结构。citeturn13view0turn13view1turn10view2turn10view3turn18search0turn2search19turn2search28turn14search16turn9search4

**首版编辑器模块**

| 模块 | 职责 | 输入 | 输出 |
|---|---|---|---|
| 场景层 | 3D 展示、选中、高亮、预览拖拽 | `SceneViewModel`、选择状态 | `pick/select/hover` 事件 |
| 结构树 | 展示模块 / 层 / 构件层级 | `resolvedDsl` 展平树 | 树选中、批量选中 |
| 属性面板 | 受约束参数编辑 | 当前选中对象 + 派生属性 DTO | 语义命令 |
| 聊天输入区 | 自然语言生成 / 局部编辑入口 | 文本、ReferenceContext | AI tool call 请求 |
| 确认层 | 展示并收集未确认项 | `confirmationDsl` | 确认命令 |
| 检测面板 | 问题列表、定位、修复建议 | `CheckIssue[]` | 聚焦、修复命令 |
| BOM 面板 | 设计 / 交易 BOM 视图 | `designBom/tradeBom` | 导出、定位源对象 |

**通信方式**

1. Chat / 属性面板 / 检测面板都只发**命令**给 `projectStore`。
2. `projectStore` 应用命令后，触发规则全量重算：`resolvedDsl + checks + BOM`。
3. `renderStore` 基于旧 / 新 `resolvedDsl` 计算 `SceneViewModel` diff。
4. `SceneAdapter` 根据 diff 只更新受影响 mesh bucket。
5. 场景选中和结构树选中都回流到 `selectionStore`，再供属性面板和聊天框使用。

**哪些状态进入 Pinia**

- `projectStore`：当前 `ProjectDocument`、revision、命令游标、派生摘要。
- `selectionStore`：当前选中、hover、最近引用历史、ReferenceContext。
- `chatStore`：消息流、AI 请求状态、失败 envelope。
- `uiStore`：面板开关、视角、筛选器、导出状态。
- `persistenceStore`：保存状态、快照索引、最近项目。

**哪些对象必须 `markRaw()` / `shallowRef()`**

- `Engine`
- `Scene`
- `ArcRotateCamera`
- `UtilityLayerRenderer`
- `GizmoManager` / `BoundingBoxGizmo`
- `SelectionOutlineLayer` / `HighlightLayer`
- 所有 `Mesh`、`Material`、`AssetContainer`
- 任何 Babylon `Observable` 订阅返回对象

**哪些操作必须走命令模式**

- 整体宽 / 深 / 高调整
- bay 增减
- 层增减 / 层高移动
- 后撑开关
- 型材系列替换
- 添加 / 删除横梁
- 恢复快照
- 应用局部自然语言编辑
- 自动修复某条检测问题

**选中、高亮、局部更新**

- 每个 mesh / instance 只保存最小 metadata：`nodeId`、`jointId?`、`semanticPath`、`geometryHash`。
- 选中来自 Babylon `onPointerObservable`；如果是 thin instance，使用 `pickingInfo.thinInstanceIndex` 找回业务对象。citeturn18search0turn2search28
- 高亮优先使用 `SelectionOutlineLayer`；二级强调使用 bounding box / gizmo utility layer。Babylon 的 selection outline 已支持 instanced 和 thin-instanced mesh。citeturn2search19turn14search14
- 局部更新不是“局部求解业务”，而是“全量求解业务 + 局部更新渲染”。
- `SceneViewModel` 推荐按 `geometryHash = profileSpecKey + cutLength + cutSpec + materialKey` 分 bucket；同 bucket 走 instance / thin instance，减少 draw call。

**如何避免 Babylon scene 成为业务真相源**

- Scene 只表达“如何显示”，不表达“结构是否合法”。
- gizmo 拖拽只产生 preview transform，不直接改文档。
- 指针抬起时把 preview 解释成语义命令，例如 `moveLevel(level=2, z=1100)`。
- 文档更新后重新跑规则，再由 `resolvedDsl` 反向驱动 scene。
- 任意时刻都允许“销毁 scene 后从 ProjectDocument 完整重建”，这就是你检验是否真正规避了 scene 泄漏的标准。

### 8. Babylon.js vs Three.js 决策

就这个项目，首版应直接选 Babylon.js。原因不是抽象层面的“渲染能力比较”，而是编辑器刚需能力的集成密度：Babylon 的 UtilityLayerRenderer 可以在不修改原场景状态的情况下叠加对象，gizmo 也明确运行在 utility layer 上；Selection Outline 原生支持 instanced / thin-instanced mesh；picking 对 thin instances 会返回 `thinInstanceIndex`；AssetContainer 能让资产与 scene 保持链接但不进入场景图。对照之下，Three.js 虽然有 `Raycaster`、`InstancedMesh`、`TransformControls`、`OutlinePass`、`CSS2DRenderer`，但后三者都属于 addons，`CSS2DRenderer` 还只支持 100% 浏览器 / 显示缩放。对一个以“选中、局部编辑、轮廓高亮、尺寸辅助、资产缓存”为主的受约束编辑器，这些差异会直接转化为首版成本和边界 bug 数量。citeturn14search16turn9search4turn2search19turn2search28turn2search7turn9search1turn10view11turn10view13turn15view0turn15view1turn15view2

**明确结论**

- 首版选 **Babylon.js**。
- 首版不再做 Babylon.js / Three.js 双实现抽象层。
- `packages/render-babylon` 单独成包，给二期保留替换空间，但首版不并行实现 `render-three`。

**放弃 Three.js 的主要原因**

1. 你要的不是“展示器”，而是“受约束编辑器”。
2. Three.js 的 `TransformControls`、`OutlinePass`、`CSS2DRenderer` 都是 addon，首版需要自己拼装更多交互胶水。citeturn15view0turn15view1turn15view2
3. 尺寸与辅助标注如果依赖 `CSS2DRenderer`，会受 100% zoom 限制影响。citeturn15view1
4. Babylon 在 selection / utility layer / asset container / thin-instance picking 上更贴近这类编辑器的中间件需求。citeturn14search16turn2search19turn2search28turn2search7

**首版必须用到的 Babylon.js 能力清单**

- `Scene.onPointerObservable`
- `UtilityLayerRenderer`
- `GizmoManager` / `BoundingBoxGizmo`
- `SelectionOutlineLayer` / `HighlightLayer`
- `AssetContainer` / `LoadAssetContainerAsync`
- glTF loader
- `instances` / `thin instances`
- `showBoundingBox` / `EdgesRenderer`
- Mesh picking + `thinInstanceIndex`

**即使用 Babylon.js 也必须额外封装的能力**

- `nodeId <-> mesh handle` 双向索引
- geometry bucket / instance pool
- scene diff 应用器
- drag preview 与 commit 分离
- HTML 面板与 3D picking 的冲突管理
- 基于 `resolvedDsl` 的尺寸标注 SVG 管线  
  这一部分不要寄希望于 Babylon 直接替你做完。

### 9. 局部编辑实现方案

这个模块必须按“引用解析器”来做，而不是按“聊天补丁器”来做。Babylon 已具备选中、outline、thin-instance picking 能力；AI 侧又能以 strict schema 输出 `editIntent`。真正的高风险不在文本生成，而在**把含糊指代稳定绑定到业务对象**。citeturn2search19turn2search28turn18search0turn10view0turn10view1turn16view3

#### 解析原则

- “这根”不是 mesh 引用，而是**当前选中业务节点**。
- “这一层”不是字符串匹配，而是 `levelIndex` / `semanticPath`。
- “后面那根”必须依赖**当前相机方向**与**已选锚点**。
- “这里加一根横梁”必须拥有**点击位置或已选构件上下文**，否则就追问。
- 所有局部编辑最终都必须提升为**结构语义 patch**，不能直接改单根杆件几何。

#### ReferenceContext 设计

```ts
export interface ReferenceContext {
  projectId: string
  revisionId: string

  activeSelection: Array<{
    entityType: 'structural' | 'joint' | 'module'
    id: string
    semanticPath: string
    role?: string
    axis?: 'x' | 'y' | 'z'
    center: Vec3
    bbox: { min: Vec3; max: Vec3 }
    bayIndex?: number | null
    levelIndex?: number | null
  }>

  hoverTarget: null | {
    entityType: 'structural' | 'joint' | 'module'
    id: string
    semanticPath: string
    worldPoint?: Vec3
  }

  cameraContext: {
    viewPreset: 'iso' | 'front' | 'right' | 'top'
    forward: Vec3
    up: Vec3
    right: Vec3
  }

  structureContext: {
    focusedModuleId?: string | null
    expandedSemanticPaths: string[]
  }

  recentReferences: Array<{
    id: string
    semanticPath: string
    referencedAt: string
  }>

  allowedActions: string[]
}
```

#### 对象引用解析顺序

1. **显式选中对象**：`这根 / 这一层 / 这里` 优先绑定 `activeSelection[0]`。
2. **hover 目标**：如果输入提交时存在 hover target，且表达是“这里 / 这个位置”，优先绑定 hover。
3. **结构树焦点**：如果树上焦点明确，则可作为二级锚点。
4. **语义角色匹配**：`横梁 / 立柱 / 后撑 / 脚` 先在选中模块内筛选 role。
5. **序数词匹配**：`第二层 / 最上层 / 中间那层` 通过 `levelIndex` 或排序匹配。
6. **相对空间匹配**：`前 / 后 / 左 / 右 / 上 / 下` 结合 `cameraContext` 与锚点同级对象集合求最近候选。
7. **最近一次引用**：`刚才那根`、省略主语时回看 `recentReferences`。
8. **仍有多个候选时强制追问**：不允许“猜一个”。

#### “后面那根”怎么解析

- 先找锚点：当前选中对象，若无则失败。
- 在同模块、同层、同 role 候选集中，计算每个候选相对锚点的屏幕空间投影。
- `后面` 解释为沿相机 forward 方向更远，且在屏幕上与锚点重叠度最高的候选。
- 若相机是 `iso` 且候选 > 1，要求用户切到 `front/right/top` 或二次确认。
- 这条规则必须 deterministic，不能交给 AI 自由发挥。

#### 需要哪些 UI 上下文辅助

- 当前选中对象 chips：`已选中：第 2 层后横梁`
- 当前视角标签：`当前视角：前视`
- 结构树 hover 联动高亮
- 发送前目标预览：输入框上方显示“本次将编辑：bay[1].level[2].rear.beamX”
- 歧义时弹候选列表并高亮对应对象
- “切到正视图”快捷按钮，减少前后左右歧义

#### 失败时如何追问

应用自己追问，不把追问文本完全外包给 AI。追问模板固定：

- “你说的‘后面那根’有 2 个候选：后上横梁 / 后下横梁，请选择。”
- “`这里加一根横梁` 缺少两端锚点，请先选中两根立柱，或改成 ‘在第二层前侧加一根横梁’。”
- “`这根加长到 1200` 会影响整个 bay 宽度，是否按 bay 宽度调整？”

#### 如何把 `editIntent` 转成 patch / command

1. AI 输出结构化 `editIntent`。
2. `ReferenceResolver` 把 `targetRef` 绑定成真实 `semanticPath` / `nodeId`。
3. `PatchPlanner` 把低层编辑提升成高层结构 patch。  
   例如：  
   - “这根加长到 1200” -> `resizeBayWidth(bayIndex=1, widthMm=1200)`  
   - “这一层上移 100” -> `moveLevel(levelIndex=2, zOffsetMm=100)`
4. 运行 preflight rules：验证是否可解。
5. 生成 `CommandEntry`，进入命令栈。
6. 全量重算业务结果，渲染层按 diff 更新。

**关键原则**：  
**不允许把单根 member 的端点作为最终编辑目标暴露给用户。**  
因为那会把产品带回 Web CAD。

#### 首版支持的局部编辑动作

| 动作 | 自然语言示例 | 转换后的命令 |
|---|---|---|
| 改整体宽度 | “整体改成 1200 宽” | `resizeOverall(widthMm=1200)` |
| 改整体深度 | “深度改成 600” | `resizeOverall(depthMm=600)` |
| 改整体高度 | “整体加高 200” | `resizeOverall(heightDeltaMm=200)` |
| 加一层 | “再加一层” | `insertLevel(afterLevel=top)` |
| 删除一层 | “删掉最上层” | `removeLevel(levelIndex=top)` |
| 调层高 | “第二层抬到离地 1100” | `moveLevel(levelIndex=2, zMm=1100)` |
| 加后撑 | “后面加后撑” | `toggleBrace(enabled=true, face='rear')` |
| 去后撑 | “去掉后撑” | `toggleBrace(enabled=false, face='rear')` |
| 换型材系列 | “这根换成 3030” / “这一层都换粗一点” | `replaceProfileSeries(scope=target, series='3030')` |
| 加横梁 | “这里加一根横梁” | `addBeam(anchorA, anchorB, level, face)` |
| 删除横梁 | “把后面这根横梁删掉” | `removeBeam(targetRef)` |
| 增加 bay | “左边再加一格” | `insertBay(side='left')` |

**首版明确不支持**

- 任意旋转 member
- 任意拖拽单个 joint 到空间任意位置
- 无锚点情况下“这里加一根杆”直接成功
- 复杂跨模块自然语言批处理

## BOM、导出与交付管线

### 10. BOM 与 SKU 映射

entity["company","MayTec","aluminum systems company"] 公开的 MayCad 资料显示，其产品能力集中在 3D / 2D 图纸与 distributor submission / BOM 导出，这说明“双 BOM + 导出链路 + 下游提交”是合理的产品能力参考，但你们不应把它当成技术依赖。citeturn10view14turn7search0

#### 设计 BOM 生成规则

1. 设计 BOM 的输入只允许是 `resolvedDsl`。
2. `StructuralNode` 中的 profile member 按  
   `profileSpecKey + cutLengthMm + cutSpec + finishKey`  
   聚合成 profile 类行项。
3. `JointNode` 不直接作为 BOM 行输出，而是先展开 joint recipe：  
   一个 joint -> 若干 connector / fastener / accessory 设计项。
4. 后撑、底脚、端盖等附属件也按 spec 级独立成行。
5. `sourceRefs` 必须回链到 node / joint，支持“BOM 点选回定位对象”。
6. 允许同一设计项多个来源对象汇总成一行，但必须保留 `sourceRefs` 明细。
7. 首版先不做市场化价格字段；设计 BOM 是工程设计产物，不是销售报价单。

#### 交易 BOM 生成规则

1. 交易 BOM 输入是设计 BOM，不是 `resolvedDsl`。
2. profile 类设计项优先走 supplier 配置的采购策略：  
   - `cut_service`：能买定尺加工件  
   - `stock_bar`：只能买原棒材  
   - `pack_only`：只能整包买
3. 若是 `stock_bar`，按同 spec 的总切割长度 + 损耗 + stock bar 长度做采购量计算。
4. connector / accessory 类通常按 `pack_size` 或 `pc` 映射。
5. 映射失败时生成 `mappingStatus = 'unmapped'` 并抛 `CheckIssue(C06)`。
6. 交易 BOM 不回写设计 BOM；它是下游采购视图。

#### SKU 映射流程

```text
resolvedDsl
  -> design parts expansion
  -> DesignBomItem[]
  -> supplier candidate match
  -> procurement policy apply
  -> pack/stock rounding
  -> TradeBomItem[]
```

#### 规格到 SKU 的映射表如何维护

建议 `packages/stdlib/catalog/` 下维护四类表：

- `profile_specs.json`
- `connector_specs.json`
- `supplier_policies.json`
- `sku_map.json`

最小 `sku_map` 关键列：

- `abstractFamilyKey`
- `specKey`
- `finishKey`
- `supplierId`
- `region`
- `sku`
- `procurementMode`
- `stockLengthMm`
- `packSize`
- `priority`
- `effectiveFrom`
- `effectiveTo`

#### catalog ETL 如何做最小实现

`tools/catalog-etl` 只做下面几步：

1. 读供应商 Excel / CSV。
2. 统一列名。
3. 通过 alias 表把脏名称映射到 `specKey`。
4. 输出 canonical JSON。
5. 运行 schema 校验与 fixture 对比。
6. 生成 ETL report：新增 / 删除 / 未映射 / 单位异常。

不要在首版 ETL 里做智能匹配。  
先把“人工维护 alias + 自动校验”跑顺。

#### 哪些映射逻辑必须后端真相化

以下逻辑必须后端维护，前端只读缓存：

- catalog 版本号
- supplier 优先级
- alias / 脏数据清洗规则
- pack rounding 和 stock bar 策略
- 生效区间 `effectiveFrom/effectiveTo`
- 导出 PDF / submission 时绑定的 catalogVersion

原因不是技术做不到，而是**交易映射是供应链事实，不是前端视图状态**。

#### 如何支持未来第三方配件扩展

标准库里先留扩展接口：

- `abstractFamilyKey`
- `mountInterfaceKey`
- `compatibilityRuleRefs`
- `defaultBomRecipe`
- `supplierExtensions[]`

这意味着二期新增第三方配件时，只要：

1. 增一个抽象 family，
2. 增兼容规则，
3. 增 specs，
4. 增 SKU map，

就能接进现有流水线，而不必重写 BOM 编译器。

### 11. 导出与持久化

MDN 文档给了这里非常清晰的边界：IndexedDB 可以事务化存储 structured-clone 支持的对象；`structuredClone()` 能深拷贝复杂值和循环引用，但函数、DOM 节点不可序列化；`localStorage` 则是跨会话保存的字符串键值。因此本项目应把完整项目文档、快照、命令日志放 IndexedDB，把最近项目索引和 UI 偏好放 `localStorage`。citeturn10view4turn10view5turn17view0turn17view1

#### 本地保存方案

- **主存储**：IndexedDB
- **索引与偏好**：`localStorage`
- **落盘对象**：只存 plain JSON / Blob，不存 Babylon 对象

#### IndexedDB 存什么

建议 object stores：

- `projects`：当前 head `ProjectDocument`
- `snapshots`：各版本快照 blob
- `commands`：命令日志
- `exports`：最近导出产物 blob
- `catalogCache`：当前 catalog 快照

#### `localStorage` 存什么

- `lastProjectId`
- `recentProjectIds`
- `ui.theme`
- `ui.viewPreset`
- `ui.panelLayout`
- `featureFlags`
- `draftPromptText`

不要把完整文档放进去。  
原因很简单：它是字符串存储，而且你根本不需要为了“最近项目列表”读一整个项目 JSON。

#### 版本快照如何设计

快照策略建议：

- 手动快照：用户点击“保存版本”
- 自动快照：  
  - 首次生成成功后  
  - 每 20 个命令  
  - 每次 AI 批量编辑前  
  - 导出 PDF 前
- 保留策略：  
  - 自动快照保留最近 30 个  
  - 手动快照长期保留

快照内容建议直接存**完整 ProjectDocument**，首版不要存 delta snapshot。  
项目规模小、正确性优先，别一开始就为省空间牺牲恢复复杂度。

#### PDF 导出放前端还是后端

**结论：首版放后端。**

理由：

- 模板版式需要稳定。
- 中英文字体、分页、表格换行更适合统一环境。
- 你们团队已有 1 名后端 / 全栈可以承接。
- 它不属于“主要编辑动作”，不违背前端本地建模原则。

前端兜底：

- 导出 `.a3dproj.json`
- 导出三视图 SVG
- 导出 design / trade BOM CSV
- 导出 HTML printable bundle

#### 尺寸标注视图如何复用 `resolvedDsl`

不要从 Babylon canvas 截图标尺寸。  
正确链路：

```text
resolvedDsl
  -> orthographic projector
  -> 2D edge / node graph
  -> dimension chain builder
  -> SVG
  -> PDF embed
```

首版尺寸内容：

- 总宽 / 总深 / 总高
- bay 宽
- 层高
- 关键中间支撑位置

#### 导出文件建议包含哪些格式

- `project.a3dproj.json`
- `design-bom.csv`
- `trade-bom.csv`
- `front-view.svg`
- `right-view.svg`
- `top-view.svg`
- `proposal.pdf`
- `preview.glb`  
  仅作展示产物，不是真相源

1. “做一个宽 1200、深 450、高 1800 的四层置物架。”

3. “总宽不要超过 900，两格。”
4. “整体用更粗一点的型材。”
5. “再加一层。”
6. “删掉最上层。”
7. “第二层抬高到离地 1100。”
8. “整体加高 200。”
9. “深度改成 600。”
10. “后面加后撑。”
11. “把后撑去掉。”
12. “左边再加一格。”
13. “把右边这一格删掉。”
14. “这根换成 3030。”（有明确选中）
15. “这一层都换成更粗的。”（选中层）
16. “后面那根删掉。”（要求相机 / 候选明确）
17. “这里加一根横梁。”（有 hover 或选中两立柱）
18. “这个方案哪里不合理？”
19. “导出设计 BOM 和采购 BOM。”

- 是否生成合法 `resolvedDsl`
- 是否无 blocker
- BOM 是否一致
- 关键尺寸是否正确

#### 如何评估

| 指标 | 定义 | 首版验收阈值 |
|---|---|---|
| 参数抽取正确率 | `intentDsl` 标注字段 exact match / 总字段数 | ≥ 90% |

| 设计 BOM 正确率 | 行项、数量、cutLength 与 gold 完全一致 | ≥ 98% |
| 交易 BOM 映射正确率 | SKU、采购单位、采购数量与 gold 一致 | ≥ 95% |
| 检测建议有效率 | 用户接受并执行的建议数 / 展示建议数 | ≥ 70% |

**现在该怎么验收**

- 不要先看 UI 漂不漂亮。
- 先跑 20 条 golden cases。
- 再看 5 个指标是否过阈值。
- 指标不过，禁止扩 scope。

|---|---|---|---|---|
| 技术风险 | Scene 与业务状态耦合 | 刷新后结果不一致、无法从文档重建 scene | 强制 `resolvedDsl` 驱动渲染；scene 只存 metadata | 一旦发现双写，暂停新功能，先清理 scene 反写 |
| 技术风险 | 局部编辑引用歧义过高 | “这根/那里/后面” 一次命中率持续低 | 强制 selection-first、正视图模式、候选追问 | 若第 5 周准确率 < 85%，缩 scope，只保留选中对象编辑 |
| 技术风险 | 业务增量求解过早复杂化 | 一次改动需要维护大量局部状态 | 首版全量重算业务，局部更新渲染 | 不做业务级增量求解器 |
| 技术风险 | PDF 排版不稳定 | 字体错位、分页错乱、前后端结果不一致 | 固定模板 + 后端统一渲染 | 若延期，先上 HTML printable + SVG/CSV bundle |
| 产品风险 | 用户把产品当 Web CAD | 提出“拖一根杆去任意位置”类需求增多 | 在 UI 中只暴露受约束属性与命令 | 明确首版不支持自由建模，避免偷做半套 CAD |
| 产品风险 | 确认层过长导致流失 | 生成前追问过多 | 把确认项限制在 blocker 决策 | 先用默认值 + 风险提示，减少非关键追问 |
| 数据风险 | catalog 脏数据导致映射错误 | 同 spec 对应多个 SKU 或单位混乱 | ETL + alias 表 + fixture 校验 | 若第 6 周仍不稳，首版仅上线设计 BOM |
| 数据风险 | 单位 / 地区差异导致 BOM 漂移 | 毫米 / 英寸、地区 catalog 混用 | `catalogVersion + region + unitSystem` 全持久化 | 首版只支持单地区、毫米制 |
| AI 风险 | structured output 虽合法但语义错误 | schema 通过但草案不可解 | AI 只出草案；规则层兜底拒绝 | 增加 `unsupportedRequests` 与 refusal 展示 |
| AI 风险 | tool schema 漂移 | 新增字段后老提示词 / 老解析器崩溃 | 所有 schema 版本化；prompts 跟 schema 同包发布 | 版本不兼容即拒绝请求，不做隐式兜底 |
| 供应链风险 | 交易 BOM 采购策略不稳 | stock bar / pack rounding 经常被人工改 | 采购策略后端真相化，写 fixture | 若映射准确率 < 95%，首版不开放交易 BOM 导出 |
| 供应链风险 | 单供应商依赖过重 | 用户需求跨地区 / 跨品牌 | 数据模型提前抽象 `supplierId` | 首版明确限定单供应商，不做“伪多供应商” |

**最后的止损线**

- 第 2 周末：如果 `draft -> resolved -> render` 还没打通，停止做聊天 UI，先保主链路。
- 第 5 周末：如果局部编辑正确率 < 85%，缩成“选中对象 + 结构化编辑词汇”模式。
- 第 6 周末：如果交易 BOM 映射正确率 < 95%，MVP 只上线设计 BOM。
- 第 7 周末：如果 PDF 仍不稳，MVP 用 SVG + CSV + JSON 导出替代 PDF，不延长主链路开发。
